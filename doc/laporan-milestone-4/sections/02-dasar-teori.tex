\section{Intermediate Code Generation}
Intermediate code generation adalah tahap backend compiler yang mengubah representasi pohon (decorated AST) menjadi daftar instruksi sekuensial yang dapat dieksekusi oleh mesin virtual. Tahap ini tidak lagi mempedulikan sintaks bahasa sumber, melainkan fokus pada sintesis dan komputasi. Tujuannya adalah meratakan struktur pohon yang rumit menjadi instruksi-instruksi primitif yang sangat sederhana \cite{aho2007compilers,cooper2011engineering}.

Dalam arsitektur compiler modern, pemisahan frontend dan backend sangat penting. Frontend berurusan dengan source code, token, parse tree, dan semantic analysis. Backend berfokus pada konversi decorated AST menjadi intermediate code dan eksekusi oleh interpreter. Dengan desain ini, jika suatu saat bahasa Arion memiliki sintaks baru, hanya frontend yang perlu dirombak selama decorated AST yang dihasilkan tetap sama \cite{appel1998modern}.

\section{Three-Address Code}
Three-Address Code (TAC) adalah format intermediate code yang paling luas digunakan di industri. Dinamakan ``three-address'' karena setiap baris instruksi didesain untuk sangat sederhana, di mana maksimal hanya boleh melibatkan tiga alamat memori: dua operand sumber dan satu lokasi tujuan \cite{aho2007compilers}. Struktur dasar TAC terlihat seperti berikut.

\begin{tcolorbox}[title=Struktur Dasar TAC,colback=gray!5,colframe=black!60,boxrule=0.5pt,arc=2mm]
\begin{verbatim}
hasil = operand1 operator operand2
\end{verbatim}
\end{tcolorbox}

Dalam bahasa tingkat tinggi, programmer dapat menulis ekspresi matematis yang sangat panjang dan kompleks. Namun, CPU tidak bisa menghitung semuanya sekaligus. CPU harus memecahnya menjadi operasi-operasi biner yang sederhana. TAC menyimulasikan batasan CPU ini dengan menggunakan variabel sementara, biasanya direpresentasikan sebagai \texttt{t1}, \texttt{t2}, \texttt{t3}, dan seterusnya.

Sebagai contoh, ekspresi \texttt{x := (a + b) * (c - d)} akan dipecah menjadi empat instruksi TAC.

\begin{tcolorbox}[title=Contoh TAC,colback=gray!5,colframe=black!60,boxrule=0.5pt,arc=2mm]
\begin{verbatim}
t1 := a + b
t2 := c - d
t3 := t1 * t2
x  := t3
\end{verbatim}
\end{tcolorbox}

Dengan memecah ekspresi menggunakan TAC, kompleksitas kode tingkat tinggi telah berhasil diturunkan menjadi barisan instruksi primitif yang sangat mudah dieksekusi oleh mesin.

\section{Arsitektur Mesin Eksekusi: Stack Machine}
Interpreter pada Arion Compiler bertindak sebagai virtual machine (VM) yang mengeksekusi intermediate code. VM ini tidak memiliki bentuk fisik, tetapi memiliki logika memori dan siklus kerja yang sama persis dengan CPU sungguhan.

\subsection{Siklus Fetch-Decode-Execute}
Sama seperti CPU sungguhan, interpreter membaca instruksi satu per satu menggunakan penunjuk khusus yang disebut Instruction Pointer (IP) atau Program Counter (PC). Interpreter beroperasi dalam siklus eksekusi yang terdiri dari tiga tahap utama \cite{cooper2011engineering}.

\begin{enumerate}
    \item \textbf{Fetch:} Membaca baris instruksi TAC yang saat ini ditunjuk oleh IP.
    \item \textbf{Decode:} Membedah baris tersebut untuk mengetahui perintahnya dan operand-nya.
    \item \textbf{Execute:} Melakukan aksi nyata, seperti menghitung nilai operasi matematika atau menyimpan nilai ke memori.
\end{enumerate}

Setelah tahap execute selesai, IP akan bergerak maju ke baris instruksi berikutnya, kecuali jika instruksi tersebut adalah \texttt{JMP} atau \texttt{JPC}, di mana IP akan melompat ke nomor baris yang dituju.

\subsection{Memori Eksekusi: The Stack}
Untuk bisa melakukan perhitungan dan menyimpan variabel, VM membutuhkan memori. Struktur memori yang paling vital adalah \textbf{stack}. Stack bekerja dengan prinsip LIFO (Last In, First Out). Mengapa menggunakan stack dan bukan array biasa? Jawabannya berkaitan erat dengan scope dan function call.

Setiap kali program memasuki blok kode baru atau memanggil fungsi, interpreter akan membuat kotak memori baru di dalam stack yang disebut \textbf{stack frame}. Stack frame menyimpan semua variabel lokal milik fungsi tersebut, argumen yang di-passing, serta return address. Saat fungsi dipanggil, stack frame baru di-push ke pucuk tumpukan. Ketika fungsi selesai, frame tersebut di-pop dari stack, dan semua variabel lokalnya otomatis musnah.

Setiap stack frame memiliki tiga slot header yang selalu diisi.

\begin{itemize}
    \item \textbf{Static Link (slot 0):} Menunjuk ke base pointer frame induk secara statis (berdasarkan lexical level).
    \item \textbf{Dynamic Link (slot 1):} Menunjuk ke base pointer frame pemanggil (runtime caller).
    \item \textbf{Return Address (slot 2):} Menyimpan alamat instruksi yang harus dieksekusi setelah fungsi selesai.
\end{itemize}

Slot-slot setelah header digunakan untuk variabel lokal dan operand sementara selama eksekusi.


\section{Translasi Control Flow}
Mengubah ekspresi matematika menjadi TAC relatif mudah karena polanya pasti. Tantangan terbesar adalah meratakan control flow seperti \texttt{if-else} dan \texttt{while}. Caranya adalah dengan menggunakan kombinasi label dan jump.

\subsection{Translasi IF-ELSE}
Untuk percabangan \texttt{if-else}, generator membangun struktur berikut. Kondisi dievaluasi terlebih dahulu. Jika kondisi bernilai salah, \texttt{JPC} melompat ke blok \texttt{else}. Jika kondisi benar, eksekusi jatuh ke blok \texttt{then}. Setelah blok \texttt{then} selesai, \texttt{JMP} melompati blok \texttt{else} menuju akhir.

\begin{tcolorbox}[title=Contoh TAC untuk IF-ELSE,colback=gray!5,colframe=black!60,boxrule=0.5pt,arc=2mm]
\begin{verbatim}
{ evaluasi kondisi }
JPC 0 L_ELSE
{ blok then }
JMP 0 L_END
L_ELSE:
{ blok else }
L_END:
\end{verbatim}
\end{tcolorbox}

\subsection{Translasi WHILE Loop}
Perulangan memiliki kerumitan ekstra karena setelah blok kode selesai, alur program harus ``melompat mundur'' ke atas untuk mengevaluasi ulang kondisinya.

\begin{tcolorbox}[title=Contoh TAC untuk WHILE,colback=gray!5,colframe=black!60,boxrule=0.5pt,arc=2mm]
\begin{verbatim}
L_START:
{ evaluasi kondisi }
JPC 0 L_END
{ blok body }
JMP 0 L_START
L_END:
\end{verbatim}
\end{tcolorbox}

\subsection{Translasi FOR Loop}
For loop memerlukan inisialisasi variabel loop, evaluasi kondisi batas, eksekusi body, dan increment/decrement variabel loop sebelum kembali ke evaluasi kondisi.

\begin{tcolorbox}[title=Contoh TAC untuk FOR,colback=gray!5,colframe=black!60,boxrule=0.5pt,arc=2mm]
\begin{verbatim}
{ inisialisasi variabel loop }
L_START:
{ load variabel loop }
{ load batas akhir }
{ compare: var <= end }
JPC 0 L_END
{ blok body }
{ load variabel loop }
{ load 1 }
{ add }
{ store ke variabel loop }
JMP 0 L_START
L_END:
\end{verbatim}
\end{tcolorbox}

\subsection{Translasi CASE}
Case statement memerlukan evaluasi selector untuk setiap label. Jika salah satu label cocok, eksekusi jatuh ke body branch tersebut. Setelah body selesai, \texttt{JMP} melompati sisa branch. Jika tidak ada label yang cocok, eksekusi melanjutkan ke instruksi setelah case.


\section{Runtime Vulnerabilities}
Interpreter yang tangguh tidak hanya tahu cara menjalankan kode yang benar, tetapi juga tahu bagaimana bertahan hidup ketika diberikan kode yang salah atau berbahaya. Bahasa tingkat tinggi yang berjalan di atas VM harus memiliki mekanisme perlindungan internal.

\subsection{Kerentanan pada Stack}
\begin{itemize}
    \item \textbf{Stack Overflow:} Terjadi ketika program terus-menerus menambahkan stack frame baru hingga melampaui batas memori. Kasus paling umum adalah infinite recursion. Interpreter harus mampu menghitung kedalaman stack dan melempar error jika melebihi batas maksimal.
    \item \textbf{Stack Underflow:} Kebalikan dari overflow. Terjadi ketika interpreter diinstruksikan untuk pop data dari stack yang sudah sepenuhnya kosong.
    \item \textbf{Stack Smashing:} Bentuk klasik dari buffer overflow. Data yang ditulis melebihi ukuran variabel lokal dapat menimpa return address, menyebabkan fungsi melompat ke memori yang salah.
    \item \textbf{Stack Corruption:} Terjadi ketika struktur stack kehilangan keseimbangan. Jumlah operasi push dan pop harus selalu simetris.
\end{itemize}

\subsection{Kerentanan Memori dan Akses}
\textbf{Out-of-Bounds Variable Access} adalah kerentanan umum yang terjadi ketika program mencoba mengakses indeks array di luar batas yang didefinisikan. Interpreter wajib memvalidasi setiap akses indeks dan melempar \texttt{IndexOutOfBoundsException} jika indeks tidak valid.

\subsection{Kerentanan Alur Eksekusi}
\textbf{Invalid Jump Targets} terjadi jika intermediate code generator melakukan kesalahan dan menyuruh interpreter melompat ke label yang tidak pernah ada dalam daftar instruksi. Interpreter yang tangguh harus memverifikasi semua target jump sebelum atau saat dieksekusi.

\subsection{Kerentanan Aritmatika}
\textbf{Numerical Overflow/Underflow} terjadi ketika nilai melampaui kapasitas tipe data. Misalnya, signed integer 32-bit hanya dapat menampung nilai maksimal sekitar 2,14 miliar. Jika nilai ini ditambah 1, terjadi wrap-around menjadi bilangan negatif. Interpreter harus mendeteksi operasi yang akan melampaui batas dan menghentikan program dengan \texttt{OverflowError} atau \texttt{UnderflowError}.
